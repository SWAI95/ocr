# 내부망(오프라인) 배포 가이드 — 한국어 문서 OCR 엔진

> 대상 환경: **Ubuntu 24.04.3 LTS · Python 3.12 · RTX 4090 ×2 (24GB) · CUDA 드라이버 12.9.86**
> 기존 `aicore` 환경(`/data/aicore/ax_mes_manage`)을 **복사해서 그 위에 OCR에 필요한 모듈만 추가**한다.
> **외부 API·인터넷 일절 사용 불가** → 모든 휠·모델·Ollama 바이너리를 빌드 머신에서 받아 반입한다.

이 가이드는 두 머신을 가정한다.

| 머신 | 인터넷 | 역할 |
|---|---|---|
| **빌드 머신** | ✅ O | 휠/모델/Ollama 번들 수집 (개발 5090 머신 또는 임시 온라인 PC) |
| **내부망 머신** | ❌ X | 실제 배포 대상 (4090 ×2) |

> ⚠️ **빌드 머신은 내부망과 동일해야 한다**: `linux x86_64`, **Python 3.12**, 그리고
> 가능하면 **aicore 환경의 복사본** 위에서 휠을 받는다. 그래야 "이미 깔린 건 건너뛰고
> 없는 것만" 정확히 떨어진다. Python minor 버전이 다르면(3.11 등) 휠이 안 맞는다.

### 인터넷 될 때 / 안 될 때 — 단계 한눈에

이 문서는 **인터넷 되는 곳에서 "받기"(PART 1) → 반입 → 인터넷 안 되는 곳에서 "설치·실행"(PART 2)** 순서다.
**네트워크가 필요한 작업은 PART 1에서 전부 끝낸다.** PART 2는 단 한 번도 외부에 접속하지 않는다.

| 단계 | 인터넷 | 어디서 | 무엇을 |
|---|:---:|---|---|
| 1. 환경 복사본 | ✅ 필요 | 빌드 머신 | aicore 복제 + Python 3.12 확인 |
| 2. requirements 작성 | ⬜ 무관 | 빌드 머신 | internal/constraints (이미 레포에 있음) |
| 3. 휠 수집 → `wheelhouse/` | ✅ 필요 | 빌드 머신 | PyPI·PyTorch(cu129)·Paddle(cu129) 다운로드 |
| 4. 모델 수집 → `models/` | ✅ 필요 | 빌드 머신 | Paddle·EasyOCR·PaddleOCR-VL 가중치 |
| 5. 시스템 .deb → `debs/` | ✅ 필요 | 빌드 머신 | fonts-nanum(+선택 poppler/tesseract) |
| 6. Ollama 번들 → `ollama_bundle/` | ✅ 필요 | 빌드 머신 | ollama 바이너리 + qwen2.5vl 모델 |
| 7. 반입 | — | (이관) | USB/승인 매체로 내부망 반입 |
| 8. 파이썬 설치 | ❌ **불가** | 내부망 머신 | `--no-index` 오프라인 설치 |
| 9. Ollama 설치 | ❌ **불가** | 내부망 머신 | tarball 풀기 + 모델 blob 복원 |
| 10. 검증·실행 | ❌ **불가** | 내부망 머신 | `OCR_OFFLINE=1` 스모크/서버 |

> 💡 **빠른 점검**: PART 2(8~10단계)에서 `pip`이 `--no-index`, OCR 실행이 `OCR_OFFLINE=1`
> 이면 외부 접속이 코드/플래그 차원에서 막힌다. 네트워크 케이블을 뽑은 상태로 통과하면 OK.

## 0. 핵심 전략 (왜 이렇게 하나)

방침(사용자 지정) = **torch 계열만 현재 cu129 버전으로 고정, 그 외는 전부 최신.
numpy 는 1.x → 2.x 로 상향.**

원칙:

1. **torch/torchvision 만 고정**(constraints): `torch 2.8.0+cu129` /
   `torchvision 0.23.0+cu129` — 4090 GPU 동작이 검증된 버전. 다른 패키지가 이걸
   끌어올리지 못하게 막는다.
2. **그 외는 최신**: `numpy>=2.1`, `opencv`, `pypdfium2`, `jiwer`, `pdf2image`,
   `python-Levenshtein` 및 모든 전이 의존성(`scipy`/`pandas`/`scikit-image` 등)을
   numpy 2 호환 최신으로 끌어올린다.
3. **단, OCR 엔진 코어 3종은 검증된 버전으로 핀**: `paddleocr==3.6.0`,
   `paddlex==3.6.1`, `transformers==4.57.6`. 엔진 코드가 이들 3.6.x API
   (`res.markdown` 프로퍼티, `PaddleOCRVL(vl_rec_backend="native")`, PPStructureV3
   파라미터)에 결합돼 있어 minor 업그레이드만으로도 인식이 깨질 수 있다.
   (최신으로 가려면 `internal.txt`의 `==` 핀을 떼고 엔진 동작을 재검증.)
4. CUDA 휠은 **cu129** 로 통일(4090은 cu126/128/129 모두 GPU 정상; torch가 이미
   cu129이라 cu129가 충돌 최소).
5. `paddlepaddle-gpu` 는 **cu129 휠을 `--no-deps` 로** 따로 설치.

> ⚠️ **numpy 2.x 상향의 부작용**: aicore 복사본에는 `numpy<2`를 요구하는 비-OCR
> 패키지(numba 0.61, tensorflow/keras, pmdarima 등)가 있다. numpy를 2.x로 올리면
> **이 복사본에서 그 패키지들은 깨진다.** 이 복사본은 **OCR 전용**으로만 쓰고
> (원본 aicore는 그대로 보존), 같은 복사본에서 다른 aicore 워크로드를 돌리지 말 것.
> OCR이 실제로 쓰는 `scipy`/`pandas`/`scikit-image` 는 `--upgrade-strategy eager`로
> 함께 numpy 2 호환 버전까지 끌어올린다(8-3 단계).

## 2. 빌드 머신: requirements / constraints 파일 작성

프로젝트의 `requirements/` 아래에 내부망용 두 파일을 만든다. (이미 레포에 있으면 덮어쓰기 전 확인)

> 아래 두 파일은 이미 레포에 생성돼 있다(`requirements/internal.txt`,
> `requirements/constraints.txt`). 내용 확인용으로만 싣는다.
>
> **두 가지 방식 중 선택**:
> - **aicore 복사본에 델타만 얹기** → `internal.txt` + `constraints.txt` (이 가이드 본문)
> - **전부 한 파일로 받기/깨끗한 venv** → `requirements/internal_full.txt` (torch·paddle
>   까지 자체 포함, constraints 불필요). 휠 수집/설치 명령은 파일 상단 주석 참고. 부록 B와 짝.

### 2-1. `requirements/internal.txt` — 추가/상향 대상 (torch 외 최신, numpy 2.x)

```text
numpy>=2.1,<3              # 1.x → 2.x 상향. scipy/pandas/scikit-image 등은 eager 로 동반 상향
paddleocr==3.6.0          # 엔진 코드가 3.6.x API에 결합 — 핀 유지 권장
paddlex[ocr]==3.6.1
transformers==4.57.6      # PaddleOCR-VL(paddle_vl) native backend 호환 버전
easyocr                   # 최신
opencv-python-headless    # 최신
pypdfium2                 # 최신
pdf2image                 # 최신(poppler 폴백)
jiwer                     # 최신
python-Levenshtein        # 최신
```

> fastapi/uvicorn/pydantic 등은 aicore에 이미 있고 OCR이 굳이 최신을 요구하지 않아
> 여기 넣지 않는다(있으면 그대로 사용). 굳이 최신화하려면 이름만 추가하면 된다.
> paddlepaddle-gpu 는 cu129 휠을 별도 단계에서 `--no-deps` 로 설치한다.

### 2-2. `requirements/constraints.txt` — torch 계열만 고정

```text
torch==2.8.0+cu129
torchvision==0.23.0+cu129
```

> torch 만 고정하고 나머지(numpy 포함)는 막지 않는다. → numpy/scipy/pandas 등이
> 자유롭게 최신(numpy 2 호환)으로 풀린다.

## 3. 빌드 머신: 파이썬 휠 수집 → `wheelhouse/`

cu129 인덱스에서 받는다. **세 인덱스**(PyPI + PyTorch cu129 + Paddle cu129)를 함께 건다.

```bash
cd /path/to/ocr           # 프로젝트 루트
mkdir -p wheelhouse

# (a) OCR 모듈 + 의존성 closure (torch만 고정, numpy 등은 최신으로 해석)
pip download -r requirements/internal.txt -c requirements/constraints.txt \
  -d wheelhouse \
  --index-url https://pypi.org/simple \
  --extra-index-url https://download.pytorch.org/whl/cu129 \
  --extra-index-url https://www.paddlepaddle.org.cn/packages/stable/cu129/

# (b) paddlepaddle-gpu cu129 휠은 의존성 없이 단독으로
pip download "paddlepaddle-gpu==3.3.1" -d wheelhouse --no-deps \
  --index-url https://www.paddlepaddle.org.cn/packages/stable/cu129/

# (c) torch/torchvision cu129 휠도 wheelhouse에 동봉(설치 시 이미 있으면 건너뜀,
#     없거나 재설치 필요할 때 대비). 이미 aicore에 있으면 생략 가능.
pip download "torch==2.8.0+cu129" "torchvision==0.23.0+cu129" -d wheelhouse --no-deps \
  --index-url https://download.pytorch.org/whl/cu129

echo "받은 휠: $(ls -1 wheelhouse | wc -l) 개 / $(du -sh wheelhouse | cut -f1)"
```

> 📌 **검증**: `wheelhouse/` 에 `paddlepaddle_gpu-3.3.1-...cu129...whl` 가 있는지 눈으로 확인.
> cu126/cpu 휠이 섞여 들어오면 4090에서 GPU가 안 잡힌다.

## 4. 빌드 머신: OCR/VLM 모델 수집 → `models/`

내부망에서는 모델을 받을 수 없으므로 **전부 미리 받아 번들에 포함**한다.
모델은 `models/` 아래로 떨어지고, 내부망에서 `OCR_OFFLINE=1` 이 같은 경로를 캐시 홈으로 잡는다.

먼저 빌드 환경에 OCR 패키지를 임시로 설치해야 모델을 받을 수 있다(빌드 머신만):

```bash
# 빌드 머신에서만 — 모델 다운로드용으로 OCR 패키지 임시 설치
pip install --no-index --find-links wheelhouse --no-deps --force-reinstall \
  paddlepaddle-gpu==3.3.1
pip install -U --upgrade-strategy eager --no-index --find-links wheelhouse \
  -c requirements/constraints.txt -r requirements/internal.txt
```

### 4-1. Paddle(텍스트·레이아웃·표·도장) + EasyOCR 모델

```bash
python scripts/download_models.py          # → models/paddle/official_models, models/easyocr
#   --with-formula  : 수식/차트 모델까지(번들 커짐, 계약서엔 보통 불필요)
```

수집되는 것: PP-OCRv5 한글 det/rec, PP-DocLayout, 표 구조, 도장(seal) 모델 (~1.8GB).

### 4-2. PaddleOCR-VL(0.9B) 모델 — paddle_vl / hybrid 의 오프라인 VLM

> ⚠️ `download_models.py` 는 PaddleOCR-VL을 받지 **않는다**. paddle_vl·hybrid(완전
> offline VLM 경로)를 쓰려면 아래를 **추가로 1회** 실행해 모델을 번들에 포함시킨다.

```bash
python - <<'PY'
import os
from pathlib import Path
root = Path.cwd() / "models"
os.environ["PADDLE_PDX_CACHE_HOME"] = str(root / "paddle")
os.environ["HF_HOME"] = str(root / "hf")
os.environ.setdefault("PADDLE_PDX_DISABLE_MODEL_SOURCE_CHECK", "True")
from paddleocr import PaddleOCRVL
# 인스턴스화만으로 PaddleOCR-VL-1.6-0.9B 가중치가 캐시로 다운로드된다.
PaddleOCRVL(vl_rec_backend="native", device="cpu",
            use_doc_orientation_classify=False, use_doc_unwarping=False)
print("[OK] PaddleOCR-VL 모델 다운로드 완료 →", root)
PY
```

받은 뒤 `models/paddle/` 와 `models/hf/` 크기를 확인한다(VL 모델 ~2GB 추가).

### 4-3. (선택) Tesseract 한글 데이터

tesseract 엔진을 baseline으로 쓸 때만. 시스템 .deb로 받는다(5단계).

## 5. 빌드 머신: 시스템 패키지(.deb) 수집 → `debs/`

PDF는 `pypdfium2`(내장)로 처리되어 **poppler가 필수는 아니다.** 폰트/도장 렌더링과
폴백, 그리고 (선택)tesseract용으로만 받는다.

```bash
mkdir -p debs && cd debs
# 한글 폰트(스모크/오버레이용) — 권장
apt-get download fonts-nanum
# PDF poppler 폴백 — 선택
apt-get download poppler-utils
# tesseract baseline — 선택
apt-get download tesseract-ocr tesseract-ocr-kor libtesseract5
cd ..
```

> apt-get download는 **의존성을 같이 받지 않는다.** 의존성까지 필요하면
> `apt-get install --download-only -o Dir::Cache::archives="$PWD/debs" <pkg>` 사용
> (다만 빌드 머신과 내부망의 OS 버전이 정확히 같아야 함).

## 6. 빌드 머신: Ollama 바이너리 + Qwen2.5-VL 모델 수집

내부망엔 Ollama가 없으므로 **바이너리 tarball + 모델 blob**을 통째로 가져간다.

```bash
mkdir -p ollama_bundle

# (a) Ollama 리눅스 바이너리 (amd64)
curl -L https://ollama.com/download/ollama-linux-amd64.tgz \
  -o ollama_bundle/ollama-linux-amd64.tgz

# (b) Qwen2.5-VL 모델을 빌드 머신 Ollama로 받아서 통째로 복사
#     (빌드 머신에 ollama가 없으면 위 tarball을 먼저 풀어 ollama serve)
ollama pull qwen2.5vl:7b          # 기본(권장). 작게 가려면 qwen2.5vl:3b
# 받은 모델은 ~/.ollama/models 에 content-addressed 로 저장됨 → 통째로 동봉
tar -C "$HOME/.ollama" -czf ollama_bundle/ollama-models-qwen2.5vl.tgz models
echo "Ollama 모델 번들: $(du -sh ollama_bundle/ollama-models-qwen2.5vl.tgz | cut -f1)"
```

> qwen2.5vl:7b ≈ 6GB. 4090 24GB면 7b 권장(품질↑). 두 GPU 중 1개만 써도 충분.

## 7. 내부망으로 반입할 번들 (체크리스트)

다음을 한 디렉토리(예: `ocr_offline_bundle/`)로 묶어 반입한다.

```
ocr_offline_bundle/
├── (코드 전체)              backend/ frontend/ scripts/ requirements/ 등 레포 전체
├── wheelhouse/             # 3단계 — 파이썬 휠
├── models/                 # 4단계 — paddle/official_models · easyocr · hf(VL)
│   ├── paddle/
│   ├── easyocr/
│   └── hf/
├── debs/                   # 5단계 — fonts-nanum (+선택 poppler/tesseract)
├── ollama_bundle/          # 6단계 — ollama tarball + 모델 tgz
└── requirements/
    ├── internal.txt
    └── constraints.txt
```

> `model_manifest.json`, `samples/` 도 함께. `data/outputs/` 는 비워서 가도 됨.

## 8. 내부망 머신: 설치 (인터넷 X)

### 8-1. aicore 환경 복사 → 프로젝트에 배치

```bash
# 프로젝트 위치 예: /data/ocr
mkdir -p /data/ocr && cd /data/ocr
# 반입한 번들을 여기에 펼친다(코드/wheelhouse/models/debs/ollama_bundle)

# aicore 환경을 OCR 전용으로 복사 (원본 aicore는 건드리지 않음)
# (A) conda:
conda create --name ocr --clone aicore && conda activate ocr
# (B) venv: 디렉토리 복사
cp -a /data/aicore/ax_mes_manage/aicore ./.venv
source ./.venv/bin/activate
python --version          # 3.12.x 확인
```

### 8-2. 시스템 패키지(.deb) 설치

```bash
sudo dpkg -i debs/*.deb || sudo apt-get -f install   # 폰트 등
fc-cache -f                                          # 폰트 캐시 갱신(선택)
```

### 8-3. 파이썬 휠 오프라인 설치 (없는 것만)

```bash
# (a) paddlepaddle-gpu cu129 먼저, 의존성 격리 설치
pip install --no-index --find-links wheelhouse --no-deps --force-reinstall \
  paddlepaddle-gpu==3.3.1

# (b) OCR 모듈 + numpy 2 상향 — eager 로 scipy/pandas/scikit-image 동반 상향
pip install -U --upgrade-strategy eager --no-index --find-links wheelhouse \
  -c requirements/constraints.txt -r requirements/internal.txt
```

> `-U --upgrade-strategy eager` 라서 **numpy 가 2.x로 올라가고**, OCR이 끌어오는
> scipy/pandas/scikit-image 등 전이 의존성도 numpy 2 호환 최신으로 함께 올라간다.
> `torch/torchvision` 은 constraints로 2.8.0+cu129에 묶여 유지된다. paddleocr/
> paddlex/transformers 는 internal.txt의 `==` 핀대로 검증 버전이 깔린다.
> ⚠️ 이 상향으로 복사본 안의 비-OCR 패키지(numba/tensorflow 등)는 깨질 수 있다(예상된 동작).

## 11. 운영 메모

- **2× GPU 활용**: 기본은 `gpu:0` 1장만 쓴다. 두 번째 GPU를 쓰려면
  `CUDA_VISIBLE_DEVICES=1 ...` 로 두 번째 워커/인스턴스를 띄우거나, Ollama와 Paddle을
  서로 다른 GPU에 분리(`CUDA_VISIBLE_DEVICES`)해 메모리 경합을 피한다.
- **문서 OCR 기본 설정(고정)**: 엔진 `paddle --table`(레이아웃 ON=PPStructureV3).
  레이아웃 OFF면 큰 이미지가 다운스케일돼 조밀 단락이 깨진다(doc_01 41.6%→1.4%).
- **VLM 백엔드 선택**(`OCR_HYBRID_VLM`): `paddle_vl`(완전 in-process offline, Ollama
  불필요) | `ollama_vl`(Qwen) | `auto`(Ollama 떠 있으면 Qwen, 아니면 paddle_vl).
  내부망에서 Ollama 운영 부담을 줄이려면 `paddle_vl` 고정도 좋은 선택.
- **후처리 정규화** 기본 ON(₩ 기호 복원 등). 끄려면 `OCR_NORMALIZE=off`.
- 모든 모델/휠은 번들에 포함되어 있으므로 **재배포 시 같은 번들을 복사**하면 된다
  (다운로드 단계 재실행 불필요).

## 부록 B. 폴백 — 복사본 상향이 지저분하면 전용 venv

aicore 복사본에 numpy 2 상향(eager)을 걸면 비-OCR 패키지가 깨지고 의존성이
엉킬 수 있다. 그게 부담스러우면 aicore 공유를 포기하고 **OCR 전용 깨끗한 venv**로
간다(가장 깔끔, torch만 cu129로 교체):

```bash
python3.12 -m venv .venv-ocr && source .venv-ocr/bin/activate
pip install --no-index --find-links wheelhouse -U pip setuptools wheel
# paddle cu129 먼저
pip install --no-index --find-links wheelhouse --no-deps --force-reinstall paddlepaddle-gpu==3.3.1
# torch만 cu129로 고정하고 나머진 최신:
pip install --no-index --find-links wheelhouse -c requirements/constraints.txt -r requirements/internal.txt
```
(이 방식은 numpy 1.x 잔재가 없어 eager가 필요 없다. wheelhouse는 3단계 그대로 사용.)

## 부록 C. Ollama systemd 상주(선택)

```bash
sudo useradd -r -s /bin/false -m -d /usr/share/ollama ollama 2>/dev/null || true
sudo tee /etc/systemd/system/ollama.service >/dev/null <<'EOF'
[Unit]
Description=Ollama Service
After=network-online.target
[Service]
ExecStart=/usr/local/bin/ollama serve
User=ollama
Group=ollama
Environment=OLLAMA_MODELS=/usr/share/ollama/.ollama/models
Restart=always
[Install]
WantedBy=default.target
EOF
# 빌드 머신에서 받은 모델을 ollama 사용자 홈으로 복사
sudo tar -C /usr/share/ollama/.ollama -xzf ollama_bundle/ollama-models-qwen2.5vl.tgz
sudo chown -R ollama:ollama /usr/share/ollama/.ollama
sudo systemctl daemon-reload && sudo systemctl enable --now ollama
```
